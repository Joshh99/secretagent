An analysis of the pipeline's failure traces reveals a few very specific, mechanical logic errors that can be surgically corrected without risking regressions on the main extraction pathways:

1. **The JSON Serialization Trap (Tuples & Dates)**
   The evaluator strictly expects actual tuple objects for Gestational Age (e.g., `('33 weeks', '5 days')`). When our L4 workflow successfully identifies and returns a native Python tuple, the intermediate JSON serialization boundary (between the tool and the LLM agent) converts it into a list (`["33 weeks", "5 days"]`). The agent then dutifully outputs a list, failing the exact-match typing constraint. By casting the tuple to its exact string representation *before* it leaves the Python tool, it safely traverses the JSON boundary and the evaluator's `ast.literal_eval` natively unpacks it back into a tuple.

2. **Charlson Comorbidity Index (CCI) Double-Counting**
   A previous hardcoded override in the `workflow` intercepted the CCI deterministic result and added points for age manually (`val += (age - 40) // 10`). The underlying deterministic `compute_direct` function already securely computes this internally, meaning the pipeline was actively double-counting age points and artificially inflating scores by +1 to +3 points.

3. **Framingham Risk Score Approximation**
   The pipeline manually bypassed the broken deterministic Framingham computation to the `simulate_medical_value` fallback. However, because we didn't specify a format, the CoT reasoning was rounding decimal risks (e.g., `6.254%`) to crude integers (e.g., `6%`). Injecting a specific decimal-forcing prompt instruction solves this natively.

4. **Scoring Calculator Hallucinations (APACHE II, Caprini, GBS)**
   The `analyze_scoring_conditions` reasoning stage frequently hallucinates arbitrary clinical thresholds (e.g., generating an A-a gradient for APACHE II when ABG values aren't present) because it isn't strictly bounded by the rules of the calculator. Enhancing the `$calculator_hints` to prohibit these exact known assumptions anchors the LLM to reality prior to extraction.
