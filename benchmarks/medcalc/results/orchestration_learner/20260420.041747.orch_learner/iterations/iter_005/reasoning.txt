1. **Specific failure pattern:** The LLM's `identify_calculator` step frequently selects the wrong calculator when there are multiple similar names, even when the exact calculator name is present in the question. For example, in case train.9019, the question explicitly asks for "albumin corrected delta ratio", but the LLM selects "Delta Ratio" instead of the available "Albumin Corrected Delta Ratio", leading to an incorrect formula and missing extracted fields (like albumin).
2. **How many failures share this pattern:** Several cases (Albumin Corrected Delta Ratio, Albumin Corrected Delta Gap) fail because the LLM chooses the non-albumin-corrected variant despite explicit wording in the question.
3. **Minimal change:** Add a simple Python string-matching pre-filter in `workflow` *before* calling the LLM `identify_calculator`. If the exact name of a calculator (ignoring case) appears in the question, we use it directly. We pick the longest match to ensure we get "Albumin Corrected Delta Ratio" over "Delta Ratio".
4. **Why it won't break currently working cases:** It only overrides the LLM when there is an *exact* substring match of a calculator's name within the question. Since this is the strongest possible signal of intent, it is strictly more accurate than relying on the LLM's probabilistic choice. It also saves LLM costs for these exact-match cases. If no exact match is found, it falls back to the existing LLM identification logic.
