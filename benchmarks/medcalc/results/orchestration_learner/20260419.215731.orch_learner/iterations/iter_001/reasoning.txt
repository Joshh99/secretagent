1. What SPECIFIC failure pattern did you identify?
Extraction often returns "none" for absent boolean conditions, which gets parsed as a string or float rather than `False`.
2. How many of the shown failures share this pattern?
This is a general extraction issue affecting scoring calculators where absence of a condition needs to be explicitly `False`.
3. What is the MINIMAL change that fixes this pattern?
In `_validate_extracted_values`, add `"none"` to the list of string values that map to `False`.
4. Why will this change NOT break currently working cases?
It only affects fields explicitly extracted as "none", mapping them correctly to boolean False rather than failing validation or causing a type error in computation.
