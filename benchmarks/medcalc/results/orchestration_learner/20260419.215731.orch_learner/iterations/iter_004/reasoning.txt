1. What SPECIFIC failure pattern did you identify?
The L4 pipeline computes Estimated Gestational Age accurately but returns it as a float (e.g., 33.857) instead of the required tuple of strings (e.g., `('33 weeks', '5 days')`). The chain-of-thought fallback correctly parses this into a tuple, but the faster, deterministic Python calculation pathway returns the raw float, which fails the evaluation check.

2. How many of the shown failures share this pattern?
At least 1 failure (Estimated Gestational Age, train.9393) explicitly shows this output format mismatch (`predicted=33.857, expected=('33 weeks', '5 days')`). This will also affect any other cases that ask for weeks and days but route through the exact deterministic computation.

3. What is the MINIMAL change that fixes this pattern?
In the L4 `workflow`, intercept the result returned by `compute_calculation`. If the question asks for "weeks" and "days" and the result is a float, convert the float into the required tuple format before returning.

4. Why will this change NOT break currently working cases?
This change only intercepts and reformats cases where the question explicitly contains "weeks" and "days" AND the deterministic calculator returns a float. It mirrors the exact same tuple formatting logic that already successfully exists and works in the `_reasoning_fallback` pathway. It doesn't affect extraction, logic routing, or any other numeric calculators.
