1. What SPECIFIC failure pattern did you identify?
Scoring calculators with many optional criteria (CHA2DS2-VASc, Caprini, Wells, etc.) are frequently falling back to the less accurate LLM CoT reasoning pathway instead of using the deterministic Python calculator.

2. How many of the shown failures share this pattern?
Several of the failures (CHA2DS2-VASc, Caprini, Wells, APACHE II, SOFA) are scoring systems with numerous optional parameters.

3. What is the MINIMAL change that fixes this pattern?
In `_validate_extracted_values`, the validation logic currently intentionally invalidates extractions if a calculator has >= 4 optional fields and the LLM didn't explicitly extract all of them (even if they were correctly omitted because the patient didn't have the conditions). I will remove the `len(optional) >= 4` condition, only routing to fallback if *zero* optional fields were extracted.

4. Why will this change NOT break currently working cases?
This is a pure relaxation of a fallback trigger. The Python calculators are already designed to handle missing optional fields (defaulting them to `False` or `0`). By allowing these extractions to pass validation, we send them to the highly accurate Python compute tool instead of forcing the LLM to do complex medical scoring arithmetic.
